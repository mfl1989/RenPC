# 開発標準・ワークフロー規範 (Development Standards)

## 1. Git 運用ルール (Git Workflow)
- **ブランチ戦略 (GitHub Flow)**
  - `main`: 本番環境用ブランチ。直接のコミットは絶対禁止。常にデプロイ可能な状態を保つ。
  - `feature/xxx`: 新機能開発用ブランチ（例: `feature/user-registration`）。
  - `fix/xxx`: バグ修正用ブランチ（例: `fix/order-calculation`）。
- **コミットメッセージ規約 (Conventional Commits)**
  - `feat`: 新機能の追加
  - `fix`: バグ修正
  - `docs`: ドキュメントのみの更新
  - `refactor`: バグ修正や機能追加を含まないコードの書き換え
  - `test`: テストコードの追加・修正
  - フォーマット例: `feat(order): 回収申込の事前見積もり機能を追加`

## 2. API 通信標準 (API Communication Standard)
フロントエンドとバックエンド間のデータやり取りは、以下のJSONフォーマットで完全に統一する。

### 成功時のレスポンス (Success)
` ` `json
{
  "code": 200,
  "message": "成功",
  "data": {
    // 実際のDTOやリストデータが入る
  }
}
` ` `

### エラー時のレスポンス (Error)
業務エラーやバリデーションエラーが発生した場合。スタックトレースはフロントに返さない。
` ` `json
{
  "code": 400, // または 401, 403, 404, 500 等の適切なHTTPステータス相当のコード
  "message": "入力された郵便番号の形式が正しくありません。", // ユーザーに直接表示可能な日本語メッセージ
  "data": null
}
` ` `

## 3. 日付とタイムゾーン (Date & Timezone)
日本国内向けのサービスであるため、タイムゾーンの取り扱いに注意する。
- **サーバー設定**: `application.yml` 等でタイムゾーンを `Asia/Tokyo` (JST) に強制指定する。
- **Javaバックエンド**: 旧API (`Date`, `Calendar`) の使用は禁止。必ず `java.time.LocalDateTime` または `java.time.ZonedDateTime` を使用する。
- **フロントエンド (表示)**: `YYYY年MM月DD日 HH:mm` のような日本向けのフォーマットで表示すること。

## 4. フロントエンド開発標準 (Frontend Standards - React & TS)
- **型定義 (TypeScript)**: `any` 型の使用は原則禁止。バックエンドのDTOに対応するインターフェースを必ず定義する。
- **スタイリング**: Tailwind CSS のユーティリティクラスを使用し、インラインスタイル (`style={{}}`) は避ける。
- **コンポーネント設計**: 1ファイルにつき1コンポーネントを原則とし、再利用可能なUIパーツは `components/ui/` ディレクトリに分離する。

## 5. 環境分離 (Environment Isolation)
- `Local`: 開発者のローカル環境。
- `Staging`: 動作確認・テスト用環境。
- `Production`: 本番環境。
※ DBパスワードや外部APIのシークレットキーは絶対にコードベース（`application.yml`など）にハードコーディングせず、環境変数 (`${ENV_VAR}`) から読み込むこと。

## 6. ログ出力標準 (Logging Standards)
バックエンド（Spring Boot）のログ出力に関する規約。

- **フレームワーク**: SLF4J + Logback を使用する（Spring Boot標準）。Lombokの `@Slf4j` アノテーションを活用する。
- **ログレベルの使い分け (Log Levels)**:
  - `ERROR`: システムエラー、DB接続エラー、予期せぬ例外。**必ずスタックトレースを出力すること**（例: `log.error("システムエラー発生", e);`）。
  - `WARN`: 業務エラー（バリデーションエラーなど）、不正なアクセス、リトライ可能な障害。
  - `INFO`: アプリケーションの起動、重要な業務処理の節目（例：申込完了時のオーダーID発行、ステータス変更）。冗長なログは避ける。
  - `DEBUG`: 開発時のトラブルシューティング用（APIのRawリクエスト/レスポンスなど）。本番環境（Production）では出力しない。
- **個人情報保護 (PII Masking) 🌟最重要🌟**:
  - ユーザーの氏名、電話番号、住所、メールアドレス、パスワードなどの個人情報（PII）は、**絶対にログファイルに平文で出力してはならない**。
  - ログに出力する場合は、マスキング処理（例: `090-****-5678`）を行うか、出力対象から意図的に除外すること。
- **トレーサビリティ (Traceability)**:
  - 各リクエストをまたいで処理を追跡できるよう、ログフォーマットには `[traceId]` などの識別子を含める設計を推奨する。
